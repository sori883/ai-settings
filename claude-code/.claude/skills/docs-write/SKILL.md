# ドキュメント作成スキル

## ドキュメントを書くとき

### ここから始める

1. **誰のためのものか？** 読者に合わせて複雑さを調整する。難しいことを過度に単純化したり、簡単なことを過度に複雑にしない。
2. **読者は何を必要としているか？** 素早く答えにたどり着けるようにする。誰もドキュメントに必要以上に長くいたくない。
3. **自分が苦労したことは何か？** 学習時によくあった疑問に答える（ただし質問そのものは書かない）。

### 執筆プロセス

**下書き：**

- 同僚に話すように手順や説明を書く
- まず何をするかを示し、次に理由を説明する
- 見出しは要点を述べる：「ユーザー追加前にSAMLを設定する」であって「SAMLの設定タイミング」ではない

**編集：**

- 声に出して読む。自分が話しているように聞こえるか？堅すぎるなら、簡潔にする。
- 読者の直接的な助けにならないものはカットする
- 各段落に明確な目的が一つあるか確認する
- 例が実際に動作するか検証する（エラーになる例を出さない）

**仕上げ：**

- リンクは説明的にする（「こちら」は使わない）
- バッククォートはコード/変数のみ、**太字**はUI要素に使う
- アメリカ英語のスペル、オックスフォードカンマ（連続カンマ）を使用
- 画像は最小限に、対象を絞って使う

**フォーマット：**

- 編集後にファイルにprettierを実行する：`yarn prettier --write <file-path>`
- これによりすべてのドキュメント間で一貫したフォーマットが保たれる

### よくあるパターン

**手順：**

```markdown
実行：
\`\`\`
実行するコマンド
\`\`\`

次に：
\`\`\`
次のコマンド
\`\`\`

これにより最新の変更が取得されます。
```

こうではなく：「（Yの前にXを実行することを忘れないでください...）」と段落の中に埋もれさせない。

**見出し：**

- 「設定に環境変数を使用する」 ✅
- 「環境変数」 ❌（曖昧すぎる）
- 「設定に環境変数を使用する方法」 ❌（冗長すぎる）

**リンク：**

- 「[SAMLドキュメント](link)を確認してください」 ✅
- 「ドキュメントは[こちら](link)を参照」 ❌

### 注意すべきこと

- タスクを「簡単」と表現すること（読者の状況はわからない）
- Metabaseの機能について「私たち」を使うこと（「Metabase」または「それ」を使う）
- 堅い表現：「活用する」「参照する」「提供物」
- テンションが高すぎる表現：感嘆符の多用
- 説明の中にアクションを埋もれさせること
- 動作しないコード例
- 古くなる数値
