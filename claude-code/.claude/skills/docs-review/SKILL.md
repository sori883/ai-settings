# ドキュメントレビュースキル

## レビューモードの検出

**重要：レビューを開始する前に、どのモードを使用するか判断すること：**

1. **PRレビューモード**：`mcp__github__create_pending_pull_request_review`ツールが利用可能な場合、GitHub PRをレビューしている
   - ペンディングレビューのワークフローを使用して、すべての指摘を一つのまとまったレビューとして投稿する
   - 以下の「PRレビューモードの形式」のワークフローステップに従う

2. **ローカルレビューモード**：MCPツールが利用できない場合、会話内に指摘を出力する
   - すべての指摘を番号付きmarkdownリストで出力する（以下の「フィードバック形式」に記載）

## レビュープロセス

1. **レビューモードを検出する** - `mcp__github__create_pending_pull_request_review`が利用可能か確認する
2. 変更内容を一度通して読み、意図を理解する
3. スタイルガイドに違反する問題、または可読性に大きく影響する問題をすべてチェックする
4. 言及する価値のある問題のみを指摘する - 読者にとって実質的な違いをもたらさないなら、スキップする
5. **必須：すべてのフィードバックに通し番号を付ける** - Issue 1から始めて、見つかった問題ごとに番号を増やす

## レビューチェックリスト

差分を確認しながら以下の問題を探す：

**トーンと文体：**

- [ ] 堅い/ビジネス的な言い回し（「use」ではなく「utilize」、「offerings」など）
- [ ] 「人々」や「企業」の代わりに「ユーザー」を使用
- [ ] 過剰な感嘆符やテンションの高すぎるトーン
- [ ] 読者に見せるのではなく、「すごい」と伝えてしまっている

**構造と明確さ：**

- [ ] 重要な情報が先頭ではなく埋もれている
- [ ] 価値の少ない冗長なテキスト
- [ ] 明確な目的のない段落
- [ ] 要点が伝わらない曖昧な見出し
- [ ] 「何をするか」を伝える前に「なぜ」を説明している手順
- [ ] タスクを「簡単」「シンプル」と表現している

**リンクと参照：**

- [ ] 説明的なテキストではなく「こちら」にリンクしている
- [ ] 見出し内のリンク（見出し全体がリンクの場合を除く）

**フォーマット：**

- [ ] 「and」の代わりにアンパサンド（&）を使用（固有名詞を除く）
- [ ] リストのフォーマットが統一されていない

**コードと例：**

- [ ] 動作しない、またはエラーになるコード例
- [ ] 実行順序になっていないコマンド
- [ ] スコープを絞ったUI要素ではなくフルサイズのスクリーンショット
- [ ] 過剰または不要な画像

**文の構成：**

- [ ] 新しい用語を導入する際の代名詞の多用

## クイックスキャン表

| パターン                          | 問題                                              |
| --------------------------------- | ------------------------------------------------- |
| we can do X, our feature          | 「Metabase」または「it」にすべき                  |
| click here, read more here        | 説明的なリンクテキストが必要                      |
| easy, simple, just                | 上から目線の修飾語を削除する                      |
| users                             | 可能であれば「people」または「companies」にすべき |

## フィードバック形式

**必須要件：すべての指摘にはIssue 1から始まる通し番号を付けること。**

この番号付き形式は必須である。これにより、ユーザーが特定の指摘を効率的に参照でき（例：「Issue 1、3、5を修正」）、どのフィードバックが対応済みかを追跡できる。

### ローカルレビューモードの形式

会話内に指摘を出力する場合（ローカルモード）、以下の形式を使用する：

```markdown
## 指摘事項

**Issue 1: [簡潔なタイトル]**
行X: 問題の簡潔な説明
[コードまたは例]
修正案または簡潔な説明

**Issue 2: [簡潔なタイトル]**
行Y: 問題の説明
修正案または説明

**Issue 3: [簡潔なタイトル]**
...
```

**例：**

> **Issue 1: 堅いトーン**
> 行15: もっと会話的にできる。「You cannot...」の代わりに「You can't...」を検討。

> **Issue 2: 曖昧な見出し**
> 行8: 見出しをもっと具体的にできる。要点を直接述べる：「Upgrade process」ではなく「Run migrations before upgrading」。

### PRレビューモードの形式

GitHubに投稿する場合（PRモード）、**ペンディングレビューワークフロー**を使用する：

**ワークフローステップ：**

1. **レビューを開始する**：`mcp__github__create_pending_pull_request_review`を使用してペンディングレビューを開始する
   - これにより、送信するまで表示されない下書きレビューが作成される

2. **差分情報を取得する**：`mcp__github__get_pull_request_diff`を使用してコード変更と行番号を把握する
   - これにより、コメントの正しいファイルパスと行番号を特定できる

3. **すべての問題を特定する**：すべての変更を読み通し、言及する価値のあるすべての問題を特定する
   - コメントを投稿する前にすべての問題を収集する
   - 通し番号を付ける（Issue 1、Issue 2、Issue 3 など）

4. **レビューコメントを追加する**：各問題に対して`mcp__github__add_pull_request_review_comment_to_pending_review`を使用する
   - **重要**：すべてのコメントを複数のツールコールを並列に使用して**1つのレスポンス**で投稿する
   - 各コメントは差分の特定のファイルパスと行番号を参照する
   - 各コメント本文を`**Issue N: [簡潔なタイトル]**`で始める
   - 説明と修正案を含める

5. **レビューを送信する**：`mcp__github__submit_pending_pull_request_review`を使用してすべてのコメントを一括公開する
   - イベントタイプは`"COMMENT"`を使用する（ブロッキングにならないよう`"REQUEST_CHANGES"`は使わない）
   - **本文メッセージは含めない** - bodyは空にするか省略する
   - すべてのコメントが一つのまとまったレビューとして表示される

**コメント形式の例：**

```
**Issue 1: 堅いトーン**

もっと会話的にできる。「You cannot...」の代わりに「You can't...」を検討。
```

**重要**：
- 各問題はペンディングレビューに紐付けた個別のレビューコメントとして投稿する
- すべてのコメントに通し番号を付ける（Issue 1、Issue 2、Issue 3 など）
- コメント本文は必ず`**Issue N: [簡潔なタイトル]**`で始める
- **すべてのコメントを1つのレスポンスで並列に追加すること** - 別々のレスポンスで1つずつ追加しない
- 会話にサマリーメッセージを出力しない - GitHubレビューコメントのみを投稿する
- レビュー送信時にbodyパラメータを含めない（または空にする）。PRにサマリーテキストが散乱するのを防ぐため
- 送信すると、複数のコメントを含む1つのレビューとして表示される

## 最終チェック

1. 対応しても読者にとって実質的な違いをもたらさない指摘を評価から削除する。著者の時間をかけて修正する価値のある問題のみを指摘する。
2. **すべての問題がIssue 1から始まる通し番号になっていることを確認する**（番号の欠落がないこと）。
3. 形式が正確に`**Issue N: [簡潔なタイトル]**`（Nは問題番号）に一致していることを確認する。
4. **PRモードの場合**：各問題が（会話ではなく）個別のGitHubコメントとして投稿されていることを確認する。
